50  
51      /**
52  	 * The TNC Client calls this method (if supported by the TNCC and 
53  	 * implemented by an IMC) to deliver a message to the IMC. This 
54  	 * method provides several features that receiveMessage does not: 
55  	 * longer (32 bit) vendor ID and message subtype fields, a 
56  	 * messageFlags field, and the ability to specify a source IMV ID 
57  	 * and destination IMC ID.
58  	 * <p>
59  	 * The message is contained in the buffer referenced by message. 
60  	 * This parameter may be null to represent an empty message. The 
61  	 * type of the message is indicated by the messageVendorID and 
62  	 * messageSubtype parameters. The message must be from an IMV (or 
63  	 * a TNCS or other party acting as an IMV). Any flags associated 
64  	 * with the message are included in the messageFlags parameter. 
65  	 * The sourceIMVID and destinationIMCID parameters indicate the 
66  	 * IMV ID of the IMV that sent this message (if available) and the
67  	 * IMC ID of the intended recipient (if the EXCL flag is set) or 
68  	 * the IMC ID in response to whose message or messages this 
69  	 * message was sent. If the EXCL flag is set, destinationIMCID 
70  	 * MUST be either the primary IMC ID for this IMC (matching the 
71  	 * value of the TNC_AttributeID_PRIMARY_IMC_ID attribute) or an 
72  	 * additional IMC ID reserved when the IMC requested the TNCC to 
73  	 * do so by calling reserveAdditionalIMCID. If the EXCL flag is 
74  	 * not set, then destinationIMCID MAY be set to the wild card 
75  	 * TNC_IMCID_ANY.
76  	 * <p>
77  	 * If an IF-TNCCS protocol that support long types or exclusive 
78  	 * delivery is used for a connection and the IMC and TNCC implement
79  	 * receiveMessageLong, the TNCC SHOULD use this method to deliver 
80  	 * messages instead of using receiveMessage. However, if the IMC 
81  	 * does not implement receiveMessageLong, the TNCC SHOULD use 
82  	 * receiveMessage instead. Messages whose vendor ID or message 
83  	 * subtype is too long to be represented in the parameters 
84  	 * supported by receiveMessage MUST NOT be delivered to an IMC 
85  	 * that does not support receiveMessageLong. This should be fine 
86  	 * since the IMC in question wouldn't have the ability to process 
87  	 * such messages. Also, the IMC probably calls reportMessageTypes 
88  	 * instead of reportMessageTypesLong so it couldn't have expressed
89  	 * an interest in messages with long types except via wild cards. 
90  	 * Messages with flags or source IMV IDs can be handled fine using
91  	 * the receiveMessage method.
92  	 * <p>
93  	 * The IMC SHOULD send any IMC-IMV messages it wants to send as 
94  	 * soon as possible after this method is called and then return 
95  	 * from this method to indicate that it is finished sending 
96  	 * messages in response to this message.
97  	 * <p>
98  	 * As with all IMC methods, the IMC SHOULD NOT wait a long time 
99  	 * before returning from receiveMessageLong. To do otherwise would
100 	 * risk delaying the handshake indefinitely. A long delay might 
101 	 * frustrate users or exceed network timeouts (PDP, PEP or otherwise).
102 	 * <p>
103 	 * IMCs are not required to implement this method but any IMC that
104 	 * implements the IMCLong interface MUST implement this method. An
105 	 * IMC should implement this method if it wants to receive 
106 	 * messages with long types or flags or source IMV IDs. Simple 
107 	 * IMCs need not implement this method. Since this method was not 
108 	 * included in IF-IMC 1.2, many IMCs do not implement it. TNCCs 
109 	 * MUST work properly if an IMC does not implement this method. 
110 	 * Likewise, many TNCCs do not implement this method so IMCs MUST 
111 	 * work properly if a TNCC does not implement this method.
112 	 * <p>
113 	 * The IMC MUST NOT ever modify the buffer contents and MUST NOT 
114 	 * access the buffer after receiveMessageLong has returned. If the
115 	 * IMC wants to retain the message, it should copy it before 
116 	 * returning from receiveMessageLong.
117 	 * <p>
118 	 * In the messageVendorID and messageSubtype parameters, the TNCC 
119 	 * MUST pass the vendor ID and message subtype of the message. 
120 	 * These values MUST match one of the values previously supplied 
121 	 * by the IMC to the TNCC in the IMCï¿½s most recent call to 
122 	 * reportMessageTypes or reportMessageTypesLong. The TNCC MUST NOT
123 	 * specify a message type whose vendor ID is 0xffffff or whose 
124 	 * subtype is 0xff. These values are reserved for use as wild 
125 	 * cards, as described in section 3.9.1. IMCs MAY check these 
126 	 * parameters to make sure they are valid and throw an exception 
127 	 * if not, but IMCs are not required to make these checks.
128 	 * <p>
129 	 * Any flags associated with the message are included in the 
130 	 * messageFlags parameter. This may include the EXCL flag but the 
131 	 * TNCC MUST process that flag itself to ensure that a message 
132 	 * with this flag set is only delivered to the intended recipient.
133 	 * <p>
134 	 * <p>
135      * @param c the IMC connection object
136      * @param messageFlags flags associated with the message
137      * @param messageVendorID vendor ID associated with the message
138      * @param messageSubtype message subtype associated with the message
139      * @param message the message to be delivered
140      * @param sourceIMVID source IMV ID for the message
141      * @param destinationIMCID destination IMC ID for message
142      *
143      * @exception TNCException if a TNC error occurs
144      */
145 	public void receiveMessageLong(IMCConnection c,
146 								   long messageFlags,
147 								   long messageVendorID,
148 								   long messageSubtype,
149 								   byte[] message,
150 								   long sourceIMVID,
151 								   long destinationIMCID) throws TNCException;
152 }
