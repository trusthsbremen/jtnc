47  public interface IMV {
48  
49  	/**
50  	 * Initializes the IMV. All IMVs MUST implement this Method.
51  	 * The TNC Server supplies itself as a parameter
52  	 * so the IMV can call the TNCS as needed.
53  	 * <p>
54  	 * The TNC Server MUST NOT call any other 
55  	 * IF-IMV API Methods for an IMV until it has 
56  	 * successfully completed a call to the IMV's initialize
57  	 * method. Once a call to this method has 
58  	 * completed successfully, this method MUST NOT be 
59  	 * called again for a particular IMV-TNCS pair until 
60  	 * a call to the IMV's terminate method has completed 
61  	 * successfully.
62  	 * <p>
63  	 * @param tncs the TNC Server
64  	 *
65  	 * @throws TNCException if a TNC error occurs
66  	 */
67      public void initialize(TNCS tncs) throws TNCException;
68      
69      /**
70       * Closes down the IMV. The TNC Server calls this method
71       * when all work is complete 
72       * and the TNCS is preparing to shut down or when 
73       * the IMV throws a TNC_RESULT_FATAL exception. Once a call 
74       * to an IMV's terminate method is made, the 
75       * TNC Server MUST NOT call the IMV except 
76       * to call the IMV's initialize method (which may
77       * not succeed if the IMV cannot reinitialize
78       * itself). Even if the IMV returns an error from
79       * this method, the TNC Server MAY continue
80       * with its unload or shutdown procedure.
81       * <p>
82       * @throws TNCException if a TNC error occurs
83       */
84      public void terminate() throws TNCException;
85  
86      /**
87       * Informs the IMV that the state of IMVConnection c
88       * has changed to newState.
89       * The TNCConstants interface lists all the possible 
90       * values of the new state for this version of the 
91       * IF-IMV API. The TNCS MUST NOT use any other 
92       * values with this version of IF-IMV.
93       * <p>
94       * IMVs that want to track the state of network 
95       * connections or maintain per-connection data 
96       * structures SHOULD implement this method. 
97       * Other IMVs MAY implement it.  If an IMV chooses
98       * to not implement this method it MUST throw 
99       * an UnsupportedOperationException.
100      * <p>
101      * If the state is TNC_CONNECTION_STATE_CREATE, 
102      * the IMV SHOULD note the creation of a new 
103      * network connection.
104      * <p>
105      * If the state is TNC_CONNECTION_STATE_HANDSHAKE, 
106      * an Integrity Check Handshake is about to begin.
107      * <p>
108      * If the state is TNC_CONNECTION_STATE_DELETE,
109      * the IMV SHOULD discard any state pertaining
110      * to this network connection and MUST NOT pass
111      * this network connection to the TNC Server
112      * after this method returns.
113      * <p> 
114      * @param c the IMV Connection
115      * @param newState new network connection state
116      *
117      * @throws TNCException
118      */
119     public void notifyConnectionChange(IMVConnection c, long newState) throws TNCException;
120 
121     /**
122      * The TNC Server calls this method to deliver a 
123      * message to the IMV. The message is contained in the 
124      * buffer referenced by message. The type of the message is 
125      * indicated by messageType. The message MUST 
126      * be from an IMC (or a TNCC or other party 
127      * acting as an IMC).
128      * <p>
129 	 * If IF-TNCCS-SOH is used for a connection and the 
130 	 * IMV and TNCS implement receiveMessageSOH(), the 
131 	 * TNCS SHOULD use that method to deliver the contents
132 	 * of SOHReportEntries instead of using receiveMessage.
133 	 * However, if IF-TNCCS-SOH is used for a connection 
134 	 * but either the IMV or the TNCS does not implement
135 	 * receiveMessageSOH, receiveMessage SHOULD be used 
136 	 * instead. For each SOHReportEntry, the value contained
137 	 * in the first System-Health-ID attribute should be 
138 	 * compared to the messageType values previously supplied
139 	 * in the IMV's most recent call to reportMessageTypes 
140 	 * or reportMessageTypesLong. The SOHReportEntry should
141 	 * be delivered to each IMV that has a match. If an IMV
142 	 * that does not support receiveMessageSOH (or when the
143 	 * TNCS does not support that method), receiveMessage 
144 	 * SHOULD be employed. In that case, the TNCS MUST pass
145 	 * in the message parameter a reference to a buffer
146 	 * containing the Data field of the first Vendor-Specific
147 	 * attribute whose Vendor ID matches the value contained
148 	 * in the System-Health-ID. If no such Vendor-Specific 
149 	 * attribute exists, the SOHReportEntry MUST NOT be 
150 	 * delivered to this IMV. The TNCS MUST pass in the 
151 	 * messageType parameter the value contained in the 
152 	 * first System-Health-ID attribute.
153      * <p>
154      * The IMV SHOULD send any IMC-IMV messages it 
155      * wants to send as soon as possible after this 
156      * method is called and then return from this 
157      * method to indicate that it is finished 
158      * sending messages in response to this message.
159      * <p>
160      * As with all IMV methods, the IMV SHOULD NOT 
161      * wait a long time before returning from the IMV
162      * receiveMessage() method. To do otherwise 
163      * would risk delaying the handshake indefinitely. 
164      * A long delay might frustrate users or exceed 
165      * network timeouts (PDP, PEP or otherwise).
166      * <p>
167      * The IMV should implement this method if it 
168      * wants to receive messages. Most IMVs will do 
169      * so, since they will base their IMV Action 
170      * Recommendations on measurements received 
171      * from the IMC. However, some IMVs may base 
172      * their IMV Action Recommendations on other 
173      * data such as reports from intrusion 
174      * detection systems or scanners. Those IMVs 
175      * need not implement this method.
176      * <p>
177      * The IMV MUST NOT ever modify the buffer 
178      * contents and MUST NOT access the buffer 
179      * after the IMV receiveMessage() method has returned. 
180      * If the IMV wants to retain the message, it 
181      * should copy it before returning from this
182      * method.
183      * <p>
184      * The message parameter may be null to represent an empty message.
185      * In the messageType parameter, the TNCS MUST pass the type of the
186      * message. This value MUST 
187      * match one of the TNC_MessageType values 
188      * previously supplied by the IMV to the TNCS in 
189      * the IMV's most recent call to the TNCS 
190      * reportMessageTypes method. IMVs MAY check 
191      * these parameters to make sure they are valid 
192      * and return an error if not, but IMVs are not 
193      * required to make these checks.
194      * <p>
195      * @param c the IMVConnection
196      * @param messageType the message type that is being delivered to the IMV
197      * @param message the message that is being delivered to the IMV
198      *
199      * @throws TNCException
200      */
201     public void receiveMessage(IMVConnection c, long messageType, byte[] message) throws TNCException;
202     
203     /**
204      * The TNC Server calls this method at the end of an
205      * Integrity Check Handshake (after all IMC-IMV 
206      * messages have been delivered) to solicit 
207      * recommendations from IMVs that have not 
208      * yet provided a recommendation. The TNCS
209      * MUST NOT call this method for an IMV for
210      * a particular connection if that IMV has
211      * already called <code>provideRecommendation</code>
212      * on that connection since the TNCS last called
213      * <code>notifyConnectionChange</code> for that
214      * IMV with that connection. If an IMV is 
215      * not able to provide a recommendation at this 
216      * time, it SHOULD call the TNCS 
217      * provideRecommendation() method with the 
218      * recommendation parameter set to 
219      * TNC_IMV_ACTION_RECOMMENDATION_NO_RECOMMENDATION. 
220      * If an IMV returns from this method without 
221      * calling the TNCS' provideRecommendation method, 
222      * the TNCS MAY consider the IMV's Action 
223      * Recommendation to be 
224      * TNC_IMV_ACTION_RECOMMENDATION_NO_RECOMMENDATION. 
225      * The TNCS MAY take other actions, 
226      * such as logging this IMV behavior, which is erroneous.
227      * <p>
228      * All IMVs MUST implement this method.
229      * <p>
230      * Note that a TNCC or TNCS MAY cut off 
231      * IMC-IMV communications at any time for 
232      * any reason, including limited support for 
233      * long conversations in underlying protocols, 
234      * user or administrator intervention, or 
235      * policy. If this happens, the TNCS will 
236      * return TNC_RESULT_ILLEGAL_OPERATION 
237      * from sendMessage and call the 
238      * solicitRecommendation method to 
239      * elicit IMV Action Recommendations based 
240      * on the data they have gathered so far.
241      * <p>
242      * @param c IMV connection
243      *
244      * @throws TNCException
245      */
246     public void solicitRecommendation(IMVConnection c) throws TNCException;
247     
248     /**
249      * The TNC Server calls this method to notify 
250 	 * IMVs that all IMC messages received in a 
251 	 * batch have been delivered and this is the 
252 	 * IMV's last chance to send a message in the
253 	 * batch of IMV messages currently being collected.
254 	 * An IMV MAY implement this method if it wants 
255 	 * to perform some actions after all the IMC 
256 	 * messages received during a batch have been 
257 	 * delivered (using the IMV's receiveMessage(),
258 	 * receiveMessageLong() or receiveMessageSOH() 
259 	 * methods). For instance, if an IMV has not 
260 	 * received any messages from an IMC it may 
261 	 * conclude that its IMC is not installed on 
262 	 * the endpoint and may decide to call the 
263 	 * TNCS' provideRecommendation method with 
264 	 * the recommendation parameter set to 
265 	 * TNC_IMV_ACTION_RECOMMENDATION_NO_ACCESS. 
266      * <p>
267 	 * An IMV MAY call the TNCS' sendMessage(), 
268 	 * sendMessageSOH()  or sendMessageLong() 
269 	 * methods from this method. As with all IMV 
270 	 * methods, the IMV SHOULD NOT wait a long 
271 	 * time before returning from the batchEnding 
272 	 * method. To do otherwise would risk delaying 
273 	 * the handshake indefinitely. A long delay 
274 	 * might frustrate users or exceed network 
275 	 * timeouts (PDP, PEP or otherwise). 
276      * <p>
277      * @param c IMV connection
278      *
279      * @throws TNCException
280      */
281     public void batchEnding(IMVConnection c) throws TNCException;
282 }
