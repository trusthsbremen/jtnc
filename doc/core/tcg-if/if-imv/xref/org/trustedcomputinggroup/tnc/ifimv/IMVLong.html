50  public interface IMVLong {
51  
52  	/**
53  	 * The TNC Server calls this method to deliver a message to the 
54  	 * IMV. This method provides several features that receiveMessage 
55  	 * does not: longer (32 bit) vendor ID and message subtype fields,
56  	 * a messageFlags field, and the ability to specify a source IMC 
57  	 * ID and destination IMV ID.
58  	 * <p>
59  	 * The message is contained in the buffer referenced by message. 
60  	 * This parameter may be null to represent an empty message. The 
61  	 * type of the message is indicated by the messageVendorID and 
62  	 * messageSubtype parameters. The message must be from an IMC (or 
63  	 * a TNCC or other party acting as an IMC). Any flags associated 
64  	 * with the message are included in the messageFlags parameter. 
65  	 * The sourceIMCID and destinationIMVID parameters indicate the 
66  	 * IMC ID of the IMC that sent this message (if available) and the
67  	 * IMV ID of the intended recipient (if the EXCL flag is set) or 
68  	 * the IMV ID in response to whose message or messages this message
69  	 * was sent. If the EXCL flag is set, destinationIMVID MUST be 
70  	 * either the primary IMV ID for this IMV (matching the value of 
71  	 * the TNC_AttributeID_PRIMARY_IMV_ID attribute) or an additional 
72  	 * IMV ID reserved when the IMV requested the TNCS to do so by 
73  	 * calling reserveAdditionalIMVID. If the EXCL flag is not set, 
74  	 * then destinationIMVID MAY be set to the wild card TNC_IMVID_ANY.
75  	 * <p>
76  	 * If an IF-TNCCS protocol that support long types or exclusive 
77  	 * delivery is used for a connection and the IMV and TNCS implement
78  	 * receiveMessageLong, the TNCS SHOULD use this method to deliver
79  	 * messages instead of using receiveMessage. However, if the IMV 
80  	 * does not implement receiveMessageLong, the TNCS SHOULD use 
81  	 * receiveMessage instead. Messages whose vendor ID or message 
82  	 * subtype is too long to be represented in the parameters 
83  	 * supported by receiveMessage MUST NOT be delivered to an IMV 
84  	 * that does not support receiveMessageLong. This should be fine 
85  	 * since the IMV in question wouldn't have the ability to process 
86  	 * such messages. Also, the IMV probably calls reportMessageTypes 
87  	 * instead of reportMessageTypesLong so it couldn't have expressed
88  	 * an interest in messages with long types except via wild cards. 
89  	 * Messages with flags or source IMC IDs can be handled fine using
90  	 * the receiveMessage method.
91  	 * <p>
92  	 * The IMV SHOULD send any IMC-IMV messages it wants to send as 
93  	 * soon as possible after this method is called and then return 
94  	 * from this method to indicate that it is finished sending 
95  	 * messages in response to this message.
96  	 * <p>
97  	 * As with all IMV methods, the IMV SHOULD NOT wait a long time 
98  	 * before returning from receiveMessageLong. To do otherwise would
99  	 * risk delaying the handshake indefinitely. A long delay might 
100 	 * frustrate users or exceed network timeouts (PDP, PEP or 
101 	 * otherwise).
102 	 * <p>
103 	 * IMVs are not required to implement this method but any IMV that
104 	 * implements the IMVLong interface MUST implement this method. 
105 	 * An IMV should implement this method if it wants to receive 
106 	 * messages with long types or flags or source IMC IDs. Simple 
107 	 * IMVs need not implement this method. Since this method was not 
108 	 * included in IF-IMV 1.2, many IMVs do not implement it. TNCSs
109 	 * MUST work properly if an IMV does not implement this method. 
110 	 * Likewise, many TNCSs do not implement this method so IMVs MUST
111 	 * work properly if a TNCS does not implement this method.
112 	 * <p>
113 	 * The IMV MUST NOT ever modify the buffer contents and MUST NOT 
114 	 * access the buffer after receiveMessageLong has returned. If the
115 	 * IMV wants to retain the message, it should copy it before 
116 	 * returning from receiveMessageLong.
117 	 * <p>
118 	 * In the messageVendorID and messageSubtype parameters, the TNCS 
119 	 * MUST pass the vendor ID and message subtype of the message. 
120 	 * These values MUST match one of the values previously supplied 
121 	 * by the IMV to the TNCS in the IMVï¿½s most recent call to 
122 	 * reportMessageTypes or reportMessageTypesLong. The TNCS MUST NOT
123 	 * specify a message type whose vendor ID is 0xffffff or whose 
124 	 * subtype is 0xff. These values are reserved for use as wild 
125 	 * cards, as described in section 3.9.1. IMVs MAY check these 
126 	 * parameters to make sure they are valid and throw an exception 
127 	 * if not, but IMVs are not required to make these checks.
128 	 * <p>
129 	 * Any flags associated with the message are included in the 
130 	 * messageFlags parameter. This may include the EXCL flag but the
131 	 * TNCS MUST process that flag itself to ensure that a message 
132 	 * with this flag set is only delivered to the intended recipient.
133 	 * <p>
134 	 * @param c the IMV connection object
135 	 * @param messageFlags flags associated with the message
136 	 * @param messageVendorID vendor ID associated with the message
137 	 * @param messageSubtype message subtype associated with the message
138 	 * @param message the message to be delivered
139 	 * @param sourceIMCID source IMC ID for the message
140 	 * @param destinationIMVID destination IMV ID for message
141 	 *
142 	 * @throws TNCException if a TNC error occurs
143 	 */
144 	// FIXME corrected this to the super interface IMVConnection as is the case in IMCConnectionLong
145 	// Genzel 2014-09-20
146 	void receiveMessageLong(IMVConnection c,
147 							long messageFlags,
148 							long messageVendorID,
149 							long messageSubtype,
150 							byte[] message,
151 							long sourceIMCID,
152 							long destinationIMVID) throws TNCException;
153 }
